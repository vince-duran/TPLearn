// COMPLETE GUIDE: HOW TO MAKE DATA FLOW DIAGRAMS (DFD)
// ========================================

// === WHAT IS A DATA FLOW DIAGRAM? ===
// A Data Flow Diagram (DFD) is a graphical representation showing:
// - How data moves through a system
// - What processes transform the data
// - Where data is stored
// - Who provides and receives data
// 
// DFDs are essential for:
// - System analysis and design
// - Understanding business processes
// - Documentation and communication
// - Identifying system requirements

// === DFD COMPONENTS AND SYMBOLS ===

// 1. EXTERNAL ENTITIES (Squares/Rectangles)
// Symbol: [Entity Name]
// Purpose: Sources and destinations of data outside the system
// Examples: [Students], [Customers], [Suppliers], [Government]
// Rules:
// - Place outside system boundary
// - Use noun phrases
// - Represent people, organizations, or other systems
// - Cannot directly communicate with each other

// 2. PROCESSES (Circles/Bubbles)
// Symbol: (Process Number | Process Name)
// Purpose: Transform input data into output data
// Examples: (1.0 | Validate User), (2.0 | Calculate Grades)
// Rules:
// - Use verb phrases (action-oriented)
// - Number hierarchically (1.0, 1.1, 1.2, etc.)
// - Must have both inputs and outputs
// - Represent business functions or activities

// 3. DATA STORES (Open Rectangles)
// Symbol: D# | Data Store Name
// Purpose: Repositories where data is stored
// Examples: D1 | Customer Database, D2 | Product Inventory
// Rules:
// - Use noun phrases
// - Number sequentially (D1, D2, D3, etc.)
// - Represent files, databases, or any data storage
// - Data flows in and out but doesn't generate data

// 4. DATA FLOWS (Arrows)
// Symbol: -----> with label
// Purpose: Show movement of data between components
// Examples: customer_info, payment_details, grade_report
// Rules:
// - Label with noun phrases (what data moves)
// - Show direction with arrowheads
// - Cannot connect external entities directly
// - Cannot connect data stores directly

// === DFD HIERARCHY LEVELS ===

// CONTEXT DIAGRAM (Level 0)
// - Single process representing entire system
// - Shows system boundaries
// - External entities and major data flows
// - High-level overview for stakeholders

// LEVEL 1 DFD
// - Breaks down context diagram into major processes
// - Shows main functional areas
// - 5-9 processes typically (cognitive limit)
// - Balances with context diagram

// LEVEL 2 DFD
// - Decomposes Level 1 processes further
// - More detailed view of specific functions
// - Usually for complex processes only
// - Maintains parent process numbering (1.1, 1.2, etc.)

// LEVEL 3+ DFD
// - Further decomposition if needed
// - Very detailed process views
// - Used for complex systems only
// - Eventually leads to process specifications

// === STEP-BY-STEP DFD CREATION PROCESS ===

// STEP 1: UNDERSTAND THE SYSTEM
// - Study system requirements and objectives
// - Identify system boundaries (what's in/out)
// - Understand business processes and workflows
// - Gather user requirements and use cases
// - Review existing documentation and ERDs

// STEP 2: IDENTIFY EXTERNAL ENTITIES
// - List all sources and destinations of data
// - Include users, customers, suppliers, other systems
// - Group similar entities if appropriate
// - Position around system perimeter
// Questions to ask:
// - Who uses the system?
// - Who provides data to the system?
// - Who receives output from the system?
// - What other systems interact with this one?

// STEP 3: IDENTIFY MAJOR PROCESSES
// - List main business functions
// - Use action verbs (Process, Validate, Calculate, etc.)
// - Keep to 5-9 processes for Level 1
// - Ensure each process transforms data
// Questions to ask:
// - What are the main things the system does?
// - What business processes are supported?
// - How is data transformed or processed?
// - What decisions does the system make?

// STEP 4: IDENTIFY DATA STORES
// - List all data repositories
// - Include databases, files, temporary storage
// - Use noun phrases for names
// - Consider master files and transaction files
// Questions to ask:
// - Where is data stored permanently?
// - What data needs to be retrieved later?
// - What files or databases exist?
// - Where is reference data kept?

// STEP 5: IDENTIFY DATA FLOWS
// - Connect components with labeled arrows
// - Show what data moves between components
// - Use descriptive names for data
// - Avoid control flows (focus on data)
// Questions to ask:
// - What data does each process need?
// - What data does each process produce?
// - Where does each piece of data come from?
// - Where does each piece of data go?

// STEP 6: DRAW THE DFD

// === DFD RULES AND CONVENTIONS ===

// SYNTAX RULES:
// 1. External entities cannot connect directly to each other
// 2. External entities cannot connect directly to data stores
// 3. Data stores cannot connect directly to each other
// 4. All processes must have both inputs and outputs
// 5. Data flows must be labeled with meaningful names
// 6. All components must be labeled clearly

// BALANCING RULES:
// 1. Inputs to parent process = inputs to child processes
// 2. Outputs from parent process = outputs from child processes
// 3. Data stores at parent level appear in child level
// 4. External entities consistent across levels

// NAMING CONVENTIONS:
// 1. External entities: Noun phrases [Customer], [Supplier]
// 2. Processes: Verb phrases (Calculate Tax), (Validate Order)
// 3. Data stores: Noun phrases D1 | Customer File
// 4. Data flows: Noun phrases customer_info, order_details

// LAYOUT GUIDELINES:
// 1. External entities around the perimeter
// 2. Processes in center, arranged logically
// 3. Data stores between related processes
// 4. Minimize crossing lines
// 5. Keep related components close together
// 6. Use consistent spacing and alignment

// === COMMON DFD MISTAKES TO AVOID ===

// MISTAKE 1: Including Control Information
// Wrong: show_error_message, activate_process
// Right: error_details, process_request
// Fix: Focus on data, not control or sequence

// MISTAKE 2: Functional Decomposition Instead of Data Flow
// Wrong: Breaking down by organizational structure
// Right: Breaking down by data transformation
// Fix: Follow data processing, not org chart

// MISTAKE 3: Too Many Processes at One Level
// Wrong: 15+ processes in Level 1 DFD
// Right: 5-9 processes maximum
// Fix: Group related processes together

// MISTAKE 4: Unbalanced Decomposition
// Wrong: Child level has different inputs/outputs than parent
// Right: Maintain same external interfaces
// Fix: Check inputs and outputs match exactly

// MISTAKE 5: Improper Labeling
// Wrong: vague labels like "data", "information"
// Right: specific labels like "customer_name", "order_total"
// Fix: Use meaningful, descriptive names

// MISTAKE 6: Missing Data Stores
// Wrong: Processes generate data from nowhere
// Right: Show where persistent data comes from
// Fix: Add data stores for all persistent data

// MISTAKE 7: Control Flows Instead of Data Flows
// Wrong: start, stop, activate, control signals
// Right: actual data being passed
// Fix: Show data content, not process control

// === CONNECTION VALIDATION RULES ===

// RULE 1: BALANCING WITH CONTEXT DIAGRAM
// Every data flow in Context Diagram MUST appear in Level 1
// Check: All external inputs/outputs accounted for?
// Validation: Count flows in Context vs Level 1

// RULE 2: NO ORPHANED COMPONENTS
// Every process must have at least one input and one output
// Every data store must be accessed by at least one process
// Every external entity must connect to at least one process

// RULE 3: DATA FLOW CONNECTIVITY
// External Entity → Process ✓
// Process → External Entity ✓
// Process → Data Store ✓
// Data Store → Process ✓
// Process → Process ✓
// External Entity → Data Store ✗ (FORBIDDEN)
// Data Store → External Entity ✗ (FORBIDDEN)
// External Entity → External Entity ✗ (FORBIDDEN)
// Data Store → Data Store ✗ (FORBIDDEN)

// RULE 4: DATA FLOW LABELING
// Use specific, meaningful names
// Describe the actual data content
// Avoid control flow terms (start, stop, activate)
// Examples:
// Good: customer_profile, order_details, payment_confirmation
// Bad: data, information, signal, control
